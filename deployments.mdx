A deployment is one build-and-release cycle for a Frame. Every push to a tracked branch triggers one. Deploys are atomic - traffic only cuts over when the new version is ready, and the old one keeps serving until then.

## The deploy flow

1. **Trigger.** A push to your Frame's tracked branch (via GitHub webhook), a manual deploy from the dashboard, or a `POST /deployments` API call.
2. **Clone.** Nubo (or your agent, if self-hosted) pulls the latest commit.
3. **Build.** [Buildpacks](https://buildpacks.io) detect your language and produce a container image. No Dockerfile required.
4. **Push.** The image goes to our registry (managed cloud) or stays local (self-hosted agents).
5. **Release.** The new container starts, a health check confirms it's up, and traffic switches over.
6. **Notify.** The dashboard shows the deploy as `active`, logs stream live, and the previous container is torn down.

## Triggering a deploy

### On push

Any push to the branch you selected when creating the Frame kicks off a new deploy automatically. This is the default flow - commit, push, watch it go live.

### Manually

From the dashboard, hit **Deploy** on the Frame to rebuild the current branch's `HEAD`. Useful for re-running a build after changing env vars.

Via API:

```bash
curl -X POST https://shuttle.withnubo.com/v2/projects/<project_id>/spaces/<space_id>/frames/<frame_id>/deployments \
  -H "Authorization: Bearer <your-jwt>"
```

### From a specific commit

Pass a commit SHA or branch name in the `revision` field to deploy something other than the current `HEAD`:

```bash
curl -X POST https://shuttle.withnubo.com/v2/projects/<project_id>/spaces/<space_id>/frames/<frame_id>/deployments \
  -H "Authorization: Bearer <your-jwt>" \
  -H "Content-Type: application/json" \
  -d '{"revision": "a1b2c3d"}'
```

## Watching a build

The Frame's **Logs** tab streams build output live over WebSocket. You'll see buildpack detection, dependency install, asset compilation, and finally the runtime process starting.

If the build fails, the Frame stays on the previous deploy (or, for first-time deploys, stays in `error`) and the full log output sticks around for debugging.

## Rolling back

Every successful deploy is kept. To roll back, go to the Frame's **Deployments** history and redeploy an earlier one - Nubo re-releases the existing image without rebuilding.

## What counts as a successful build

By default: the buildpack finishes without error, the container starts, and your process doesn't crash within the first 10 seconds. Health checks beyond that are on your app to expose.

## Branches and Spaces

Spaces (`dev`, `staging`, `production`) give you one Frame per environment. The typical setup is:

- `dev` Frame tracks `develop`
- `staging` Frame tracks a release branch
- `production` Frame tracks `main`

Each Space has its own variables, its own domain, and its own deploy history. [Environment variables](/environment-variables) can be scoped per-Frame so `dev` and `production` don't share secrets.
