A deployment is one build-and-release cycle for a Frame. Deploys are atomic - traffic only cuts over when the new version is ready, and the old one keeps serving until then.

## The deploy flow

1. **Trigger.** A manual deploy from the dashboard, or a `POST /deployments` API call.
2. **Clone.** Nubo pulls the current revision of the tracked branch.
3. **Build.** Nubo automatically detects your language and produces a deployable artifact. No Dockerfile required.
4. **Release.** The new version starts, Nubo confirms it's up, and traffic switches over.
5. **Notify.** The dashboard shows the deploy as `active`, logs stream live, and the previous version is torn down.

## Triggering a deploy

From the dashboard, hit **Deploy** on the Frame to rebuild the current revision of the tracked branch. Useful for re-running a build after changing env vars.

Via API:

```bash
curl -X POST https://shuttle.withnubo.com/v2/projects/<project_id>/spaces/<space_id>/frames/<frame_id>/deployments \
  -H "Authorization: Bearer <your-jwt>"
```

The branch and revision come from the Frame's existing config, so no body is required.

## Watching a build

The Frame's **Logs** tab streams build output live. You'll see language detection, dependency install, asset compilation, and finally the runtime process starting.

If the build fails, the Frame stays on the previous deploy (or, for first-time deploys, stays in `error`) and the full log output sticks around for debugging.

## Rolling back

Every successful deploy is kept. To roll back, go to the Frame's **Deployments** history and redeploy an earlier one - Nubo re-releases the existing artifact without rebuilding.

## What counts as a successful build

By default: the build finishes without error, the app starts, and your process doesn't crash within the first 10 seconds. Health checks beyond that are on your app to expose.

## Branches and Spaces

Spaces (`dev`, `staging`, `production`) give you one Frame per environment. The typical setup is:

- `dev` Frame tracks `develop`
- `staging` Frame tracks a release branch
- `production` Frame tracks `main`

Each Space has its own variables, its own domain, and its own deploy history. [Environment variables](/environment-variables) can be scoped per-Frame so `dev` and `production` don't share secrets.
