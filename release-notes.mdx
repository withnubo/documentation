## Bring your own servers, push to deploy, see what's actually happening, and keep your streak alive

This is the biggest release Nubo has ever shipped, and it changes what the platform is. Self-hosted Agents open the door to running Frames on your own infrastructure, while the managed platform gets real CI/CD, preview URLs for every pull request, one-click rollback, and live visibility into what's running. On top of that, this release adds a metrics tab for every Project, a streak widget that tracks your shipping cadence, transactional email notifications, and a refreshed sidebar built around the work you actually do. No YAML, no runners, no extra services to wire up.

### Introducing self-hosted Agents

<Frame>
![Agent setup in the dashboard](/images/release-notes/04-29-2026/agent.png)
</Frame>

Need to run your workload on specific hardware, keep data on-premise, or use a box you already own? Install the [Nubo Agent](/agents/overview) on your own Linux or macOS server and deploy Frames to it from the dashboard the same way you would to our managed cloud.

The agent is a lightweight binary that connects back to Nubo and waits for work. One command to install, wildcard DNS to your server, and you're done. Your Frames get their own subdomain on a domain you control, with automatic TLS via ACME.

Everything else in this release works the same on self-hosted Agents as it does on managed Frames. Push-to-deploy, preview deployments, live status, deploy history, one-click rollback, project and frame environment variables, custom domains - all of it.

See the [Agents overview](/agents/overview) to understand when to reach for one, or jump to [Quickstart](/agents/quickstart) to register and install your first agent.

### Push to deploy

<Frame>
![Push-to-deploy flow](/images/release-notes/04-29-2026/push_to_deploy.png)
</Frame>

Install the [Nubo GitHub App](https://github.com/apps/nubo-apps) on your repo once and every push to your Frame's tracked branch kicks off a build and release. That's it. Nothing else to configure.

Need finer control?

- `[skip ci]` or `[ci skip]` in a commit message skips that push.
- **Ignored paths** (e.g. `docs/**`, `*.md`) skip builds when only matching files changed. Great for monorepos or docs-only commits.
- **Auto-deploy** can be toggled off if you'd rather trigger builds manually.

Everything is on the Frame's new **CI/CD** tab. See [CI/CD](/ci-cd) for the full tour.

### Personal access tokens for scripted clients

Anything automated that talks to Nubo - a CI runner, a deploy bot, your own scripts - should now use a **personal access token** instead of your login session. Create tokens in **Settings → Developer tokens** with a name and optional expiration, then use them as a `Authorization: Bearer nubo_pat_...` header.

Tokens are shown exactly once at creation, stored on our side as a hash (never in plaintext), and can be revoked instantly from the same settings page. Give each integration its own token so a leak or team change only affects that one scope of work. See [API authentication](/api-reference/authentication) for the full flow.

### Date-based API versioning

Pin your API client to a specific contract with the `Nubo-Version` header:

```
Nubo-Version: 2026-05-08
```

- **Most releases ship under the same date.** Bug fixes, new endpoints, and tuning don't move your pin. A new version is only cut when a change would actually break existing clients (field removed, type changed, validation tightened).
- **Unpinned clients get the oldest supported version**, so a new version never silently upgrades your integration.
- **Responses include a `Nubo-Build` header** naming the exact build that served them, useful for confirming what your client actually got.
- **Old versions overlap new ones** for at least one release cycle before being retired.

Production integrations should pin. Scripts and one-offs can skip the header and ride the default. See [Stability policy](/api-reference/stability) for the full rules, and [Authentication](/api-reference/authentication) for where it fits in a request.

### Preview deployments for every PR

<Frame>
![Preview deployment URL in the dashboard](/images/release-notes/04-29-2026/preview_deployments.png)
</Frame>

Every pull request opened against your tracked branch now gets its own running copy of the app at a Nubo-assigned URL. Share with teammates, click around, show a customer what's coming, and know that what they see is exactly what will ship.

Previews rebuild on every push to the PR and tear themselves down automatically when the PR closes. Project-scoped [environment variables](/environment-variables) carry over, so secrets and config work out of the box. See [Preview deployments](/preview-deployments).

### One-click rollback

<Frame>
![Rollback button in the dashboard](/images/release-notes/04-29-2026/rollback.png)
</Frame>

Shipped a bad commit? Pick any previous working deploy from the Deployments tab and hit **Rollback**. Your Frame is back on the good version within seconds, no rebuild needed. Works identically on our managed cloud and on self-hosted Agents. See [Rollback](/rollback).

### Cancel a deploy in progress

Pushed the wrong commit, or watching a build chew through what you already know is going to fail? Hit **Cancel** on a still-building deploy and Nubo stops it right where it is. Your Frame stays on whatever was already running, so there's no harm in pulling the cord. See [Cancel a deploy](/deployments#cancel-a-deploy-in-progress).

### Stop and start a Frame

Need to take a Frame offline for a bit without losing its config, deploy history, or domain? Hit **Stop** to pause it (replicas scale to zero, or the agent container is stopped on a self-hosted setup) and **Start** to bring it back up later. Everything else about the Frame is kept exactly where you left it. See [Stop and start a Frame](/frames#stop-and-start-a-frame).

### Live Frame status

<Frame>
![Live Frame status in the dashboard](/images/release-notes/04-29-2026/runtime_status.png)
</Frame>

Your Frame's status now updates in the dashboard the moment it changes. No refresh, no five-second poll. If an app crashes, runs out of memory, or fails its health check, the dashboard flips to an error state and surfaces the reason right there. You're not digging through logs just to figure out something's wrong.

### A real deploy history

<Frame>
![Deployment history](/images/release-notes/04-29-2026/deployment_history.png)
</Frame>

Every build is now a first-class record. Open the Frame's **Deployments** tab and you'll see the commit, the time, the trigger source, and whether it succeeded. Pull the same data programmatically:

```bash
curl "https://shuttle.withnubo.com/v2/projects/<p>/spaces/<s>/frames/<f>/deployments?limit=20" \
  -H "Authorization: Bearer <your-token>"
```

### Build logs, captured on every build

Every build's full output is captured the moment the build finishes, whether it succeeded or failed, so postmortems and flaky-build comparisons just work. Build logs are kept for 7 days, and the latest deploy of each Frame is always kept on top of that, so you'll never lose your most recent log even if the rest of the window has aged out. Past 7 days we return a clear "expired" signal so you can tell aged-out logs apart from a genuine fetch error.

### Live runtime logs

The Frame's Logs tab now streams live lines from your running app, not just build output. Open the tab while a Frame is online and you'll see what your code is printing right now, with no refresh and no log file path to remember. Multiple tabs subscribed to the same Frame all see the same stream, and the line ordering matches what your app actually emitted.

### Custom domains for any Frame

Bring your own hostname. Add a custom domain from the Frame's Domains tab, point your CNAME or A record where Nubo tells you to, and we handle TLS end to end. Domains are managed per-Frame, so the same Project can serve `app.example.com` from production and `staging.example.com` from your staging Space without any config plumbing. Works the same way on self-hosted Agents under the wildcard you configured. See [Custom domains](/custom-domains).

### Auto-pause for crashlooping Frames

If a Frame's container crashes or runs out of memory five times in a row, or stays stuck pulling its image for more than three minutes, Nubo automatically scales it to zero and surfaces a `backoff_tripped` reason on the Frame's status. The next push (or hitting Deploy) scales it back up, so a fresh image automatically clears the trip. You no longer have to babysit a stuck Frame: the platform takes it out of the loop until you ship a fix.

### A metrics tab for every Project

Open any Project and you'll find a new **Metrics** tab in the header. It's the answer to "how is this thing actually doing?":

- **Health at a glance**: total Frames, how many are online, how many need attention.
- **Live CPU and memory**: the last 7 days plotted as smooth area charts so trends are obvious.
- **Build performance**: p50, p95, and average build time over the last 14 days. Builds creeping up? You'll see it.
- **Per-Frame breakdown**: replicas, domains, deploy count, failure count, mean time to recover from a bad deploy, and the time of the last deploy. Sort by anything.
- **A 14-day deploy bar chart** with success and failure layered, so a flaky run sticks out.

The Project page also gets a compact preview of the metrics right under the architecture, so you don't have to switch tabs to take the temperature.

### Build a streak

The sidebar has a new activity widget at the bottom. It tracks how many days in a row you've shipped something and rewards you with milestones as you climb:

- **3 days**: heating up
- **7 days**: on fire
- **14 days**: blazing
- **30 days**: diamond
- **50 days**: legend
- **100 days**: GOAT

Each tier recolors the streak counter, the bars under it, and a glowing under-bar that fuses your streak together. We will not pretend this isn't designed to make you want to deploy on a Sunday. It is.

Below the streak there's a 60-day bar chart of your daily deploys so you can see the pattern of your work over a couple of months.

### Email notifications

We now send transactional emails for the things you actually want to know about:

- **Welcome** when you sign up.
- **Replies on your support tickets**, and a heads-up to our team when you open one.
- **Account state** changes (suspended, reinstated).
- **Build or deploy failures** on your Frames, with the last 30 lines of the build log right in the email so you don't have to log in to see what blew up.
- **Custom domains** when they're connected or removed.

You can reply to any of them and reach a human at `support@withnubo.com`.

### Frame controls in the sidebar

Click any Frame and you'll see a new control strip at the top of the sidebar:

- **Redeploy**: kick a fresh deploy from the latest commit on the tracked branch.
- **Stop / Start**: toggle replicas to zero and back without leaving the page.
- **Open**: jump straight to the Frame's primary domain.

No more bouncing through Settings to do a redeploy.

### Pull from your organization repos

If you'd installed the Nubo GitHub App on an organization but only saw your personal repos when creating a Frame, that's fixed. The repo picker now spans every installation you have access to on GitHub - personal account first, then each org you belong to. Repos with the same name across different owners are now keyed by full path, so no more collisions.

### A sidebar that gets out of your way

<Frame>
![Collapsed sidebar with activity heatmap and account dropdown](/images/release-notes/05-08-2026/collapsed_sidebar.png)
</Frame>

Hit the new chevron in the sidebar header to collapse it to a thin icon-only strip. Each Project shows up as a colored tile with its initials so you can still pick one without expanding. The notification ping moves onto your avatar at the bottom-left so you don't miss anything. Click the avatar to get a small Profile / Sign out menu in either state.

### A visual refresh

The architecture canvas, Frame tiles, Frame sidebar, and metrics view have all been pulled into the same console-style design language as the home dashboard - flat surfaces, mono captions, clean status pills. No more status-tinted gradient backgrounds; status now lives in a small dot and a label, so failed Frames stay easy to spot without painting half the sidebar red.

### Other fixes

- The "New Frame" repo picker no longer flickers and re-fetches every 15 seconds while it's open.
- The Frame status dot now correctly shows red for `failed` Frames (was reading as the neutral "unknown" color).
- Deploy timestamps in the metrics view render as readable dates instead of "Invalid Date".
- The architecture canvas is now bounded in height and scrolls inside its own box, so a project with many Frames doesn't push everything else off the page.
- Build logs panel is capped at a sensible height so a chatty deploy doesn't make the sidebar run the full viewport.
- Updating a Frame's settings no longer affects unrelated fields.
- Fetching a single Frame via the API now works correctly under the full path.
